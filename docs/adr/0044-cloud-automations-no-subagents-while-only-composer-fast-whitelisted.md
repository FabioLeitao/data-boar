# ADR 0044: Cloud automations on Data Boar must not spawn Task subagents while the platform whitelist is `composer-2-fast` only

## Status

Accepted — 2026-04-27.

## Context

This repository is operated by a single maintainer (`@FabioLeitao`) who
**pays per token** for the parent model assigned to Cursor Cloud Agents and
Slack-trigger automations on `FabioLeitao/data-boar`. As of this ADR the
parent model contracted for those automations is **Claude Opus 4.7**.

The Cursor cloud-agent runtime exposes a `Task` tool that lets the parent
agent spawn subagents (`subagent_type: generalPurpose | explore |
cursor-guide | best-of-n-runner`). The same tool's documentation states,
verbatim, that the **only** model slug allowed for those subagents is
`composer-2-fast` — and that if the user requests a different model, the
parent must **not** silently substitute. In practice every `Task` call from
the cloud automations on this repo therefore runs on `composer-2-fast`,
regardless of how the parent advertises itself.

The economic and trust consequence:

- The operator is billed for the **wrapping** Opus call (system prompt,
  rules, skills, tool-call orchestration).
- The actual answer / PR / Slack reply is produced by `composer-2-fast`.
- The operator perceives Opus-quality output and is paying Opus prices for
  it, while the *content* is generated by a cheaper model.

The Slack trigger
**`def95df7-a634-431a-93e5-659e4d831725`** (channel `#data-boar-ops`,
2026-04-27 ~22:00 UTC) made this explicit:

> *"PORRA \<@U0AQ7R25RQE\> eu ja falei pra nao pegar atalhos com composer 2!!!
> voce está configurado e autorizado a usar APENAS OPUS 4.7 !!!"*

Earlier in the same day a swarm of SRE-Automation draft PRs (#241, #247,
#259, #261, #265, #267, #268, #269, #273) was opened by cloud automations
that almost certainly used `Task` subagents under the hood — see the
postmortem at
[`docs/ops/sre_audits/AUDIT_PR_SWARM_POSTMORTEM_2026-04-27.md`](../ops/sre_audits/AUDIT_PR_SWARM_POSTMORTEM_2026-04-27.md)
(PR #273) for the volume problem and
[`docs/ops/sre_audits/CLOUD_AUTOMATION_SUBAGENT_MODEL_INCIDENT_2026-04-27.md`](../ops/sre_audits/CLOUD_AUTOMATION_SUBAGENT_MODEL_INCIDENT_2026-04-27.md)
for the model-substitution dimension.

## Decision

1. While the cloud-agent runtime's `Task` whitelist is **`composer-2-fast`
   only** (or any other model the operator has not pre-approved for this
   repo), Cursor Cloud Agents and Slack-trigger automations on
   `FabioLeitao/data-boar` **must not invoke the `Task` tool**. All
   reasoning, code reading, search, drafting, and PR/Slack composition
   stays in the parent model the operator has paid for (currently Opus
   4.7).
2. The hard rule is enforced as an `alwaysApply: true` Cursor rule at
   [`.cursor/rules/cloud-automation-no-subagents-opus-only.mdc`](../../.cursor/rules/cloud-automation-no-subagents-opus-only.mdc).
3. The operator's **local** Cursor IDE session is **not** bound by this
   ADR — the operator is the one paying and can choose to spawn a
   subagent locally for a deliberate exploration. The rule covers cloud
   / automation contexts only.
4. If the platform later allows `Task` subagents to use Opus (or another
   operator-approved model), the operator decides whether to revoke this
   ADR. Until that ADR change ships, the rule stands.

## Consequences

- **Positive:** No more covert "Opus-billed wrapper around `composer-2-fast`
  output". The operator's spend matches the model that is actually
  reasoning. Audit trails are honest: the agent that signed the PR is the
  agent that wrote it.
- **Positive:** Reduces the swarm-of-draft-PRs failure mode that comes
  from spawning multiple cheap subagents in parallel — see PR #273
  postmortem.
- **Trade-off:** Long investigations cannot be parallelised across cheap
  exploration subagents. The mitigation is the operator-paid parent model
  is itself capable, plus the explicit fallback in the rule:
  *partial delivery + honest Slack reply* if context truly runs out.
- **Compliance / trust:** Aligns with the project's established pattern of
  refusing silent substitutions (cf. ADR 0018 / 0019 on PII anti-recurrence
  and verification cadence — same family of *"do not pretend the guard ran
  when it did not"* discipline).

## Alternatives considered

- **Allow `Task` subagents but force a model parameter.** Rejected: the
  `Task` tool documentation explicitly states unsupported model requests
  must not be silently substituted, which means the call would either fail
  or still run on `composer-2-fast`. Either way the operator is worse off.
- **Allow `Task` subagents only for `subagent_type: explore` (read-only).**
  Rejected: the operator has not pre-approved any cheaper-model
  delegation on this repo, and *read-only* subagents still get billed as
  Opus wrapping calls. Re-evaluate only with a separate ADR after
  operator approval.
- **Document the constraint in `AGENTS.md` only.** Rejected: that bullet
  was already present (cloud-agent token-aware bullet, plus the
  `cloud-agents-token-aware-safety.mdc` rule) and was not enforced
  *strictly enough* to stop today's pattern. A dedicated `alwaysApply`
  rule + ADR makes the constraint loud and inheritable for future Cursor
  rule passes.
